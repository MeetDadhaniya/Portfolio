## 1. Overview

**MEET** is the personal developer portfolio for a full-stack Python/React engineer
("meet"), built as a single-page Next.js application framed entirely as the UI of a
tactical-shooter video game. Its primary visitors are recruiters, hiring managers,
fellow engineers, and potential collaborators evaluating meet for paid work — and the
site's job is to make that evaluation memorable while still surfacing real engineering
substance.

**Creative concept:** the entire portfolio is staged as a game's HUD. Visitors land on
a "Main Menu" title screen, "queue" into the site via a matchmaking loading bar, and
read each portfolio section as an in-match overlay (Agent Bio = the agent select screen,
Agent Roster = playable agents = projects, Loadout = equipped gear = skills, Match
Summary = post-match scoreboard = contact). Visuals are intentionally minimal: flat
pixel-art accents and crisp 1px HUD borders over a near-black charcoal background, with
a single Valorant-style red (#FF4655) accent. No 3D, no photographic textures, no heavy
retro pastiche — pixel-art is used sparingly for charm; layout/typography stays clean
and readable.

### Per-screen requirements

#### 4.1 Main Menu (`/`, stage: `menu`)
- **Purpose:** Set the tone and let the visitor "queue in".
- **Must contain:**
  - "MEET" pixel-font title.
  - One-line tagline: stack + agent file number.
  - Three menu items: **PLAY**, **SETTINGS**, **EXIT** (with hotkey labels 01/02/03).
  - HUD top bar (version, region/ping/tick) and bottom bar (status, online indicator).
  - Ambient layers: scrolling system log (bottom-left), background birds (top region),
    `BIRDS NN` counter.

#### 4.2 Main Menu — Settings (`/`, stage: `settings`)
- **Purpose:** Surface two toggles without leaving the menu.
- **Must contain:** REDUCED MOTION toggle, SOUND FX toggle, BACK button.

#### 4.3 Main Menu — Matching (`/`, stage: `matching`)
- **Purpose:** Brief, themed transition between menu and site.
- **Must contain:** "FINDING MATCH..." text, animated loading bar (1.4s), queue label,
  ETA label.
- **Hard requirement:** total visible duration must be ≤ 2s before site content appears.

#### 4.4 Agent Bio (`#about`)
- **Purpose:** Who is meet, in tactical-agent voice.
- **Must contain:**
  - Codename, role tag, agent portrait (pixel-art helmet).
  - Three stats (AGE / ORIGIN / RANK).
  - Lore paragraph (~3–5 sentences).
  - Four abilities (Q / E / C / X) with one-line descriptions each — these are meet's
    soft skills reframed as abilities.

#### 4.5 Agent Roster (`#work`)
- **Purpose:** Project showcase as a roster grid.
- **Must contain:** 3-up on desktop / 2-up on mobile grid of project cards. Each card:
  pixel sprite, agent ID, name, role tag, one-line hook, "LOCK IN ▶" hover hint.
  Clicking a card navigates to `/agents/[slug]`.

#### 4.6 Loadout (`#skills`)
- **Purpose:** Equipped tech stack at a glance.
- **Must contain:** Three columns:
  - **PRIMARY** (main combat kit — top 3–5 tools, each with a one-liner note).
  - **SECONDARY** (supporting tools — flat list).
  - **UTILITY** (passive traits = soft skills / workflow habits).
- Each item carries a tier letter (S / A / B), color-coded.

#### 4.7 Match Summary (`#contact`)
- **Purpose:** End-of-experience CTA and contact funnel.
- **Must contain:**
  - "VICTORY" banner.
  - "MATCH MVP" badge for meet with score-style stats (ACS / KDA).
  - Six-stat scoreboard (projects shipped, years in game, etc.).
  - Four "ADD FRIEND" social buttons (GitHub, LinkedIn, Twitter/X, Email).
  - Primary "PLAY AGAIN ▶" button → mailto:.

#### 4.8 Agent Detail (`/agents/[slug]`)
- **Purpose:** Long-form per-project page.
- **Statically generated** at build time via `generateStaticParams`.
- **Must contain:**
  - Sticky header with "◀ BACK TO ROSTER".
  - Hero: pixel sprite, agent ID, YEAR / ROLE / STATUS stats, large glitched title,
    role line, prose description, highlight chips, primary CTA (DEPLOY / INSPECT REPO
    / "// CLASSIFIED ACCESS" placeholder).
  - Abilities panel (Q/E/C/X) describing core features.
  - Equipped Stack panel with animated proficiency bars (0–100%) per technology.
  - "Patch Notes" line (year · status · maintained by AGENT.MEET).
  - "SWITCH AGENT" footer grid with up to 3 other projects.

## 5. Functional Requirements

> **FR-N.M** = Feature Number . Sub-requirement

### FR-1 Custom Crosshair Cursor
- **FR-1.1** Replace native cursor with a minimal red cross (4-arm, center gap).
- **FR-1.2** States:
  - Default: scale 1.0.
  - Hovering an interactive element (`a / button / [role="button"] / input / textarea /
    select / [data-interactive]`): scale 1.4.
  - Mousedown: scale 0.7 and show a 1px-bordered 18×18 square around the crosshair.
- **FR-1.3** Render with `mix-blend-difference` so it stays visible on light + dark
  backgrounds.
- **FR-1.4** **Touch fallback:** on `(hover: none), (pointer: coarse)` devices the
  component returns `null` and native cursor is restored.

### FR-2 Mini-Map Navigation
- **FR-2.1** Fixed bottom-right widget, only visible after the user has entered the site
  (post-PLAY).
- **FR-2.2** Square HUD frame with 4 dots representing About / Work / Skills / Contact at
  fixed percentage positions inside the box.
- **FR-2.3** Active section dot is filled with accent color and pulses; inactive dots
  use foreground color.
- **FR-2.4** Active section is detected via `IntersectionObserver` (threshold 0.35) on
  the section IDs.
- **FR-2.5** Clicking a dot smooth-scrolls to that section via Lenis
  (`duration: 1.2s, offset: -20`).
- **FR-2.6** **Mobile:** must not occlude primary CTAs on viewports < 640 px wide
  (see §8 issue I-9 — currently it does).

### FR-3 Round Transition
- **FR-3.1** Each of the four main sections is wrapped in a `RoundTransition` that
  triggers once on first view (`IntersectionObserver` threshold 0.18).
- **FR-3.2** On trigger: full-viewport overlay with a red horizontal bar wiping in
  from the left and out to the right (1.0s total). Two thin white sweep lines
  reinforce the wipe.
- **FR-3.3** Bar contains: checkpoint code (left), pixel-font label (center),
  "ROUND START" (right, desktop only).
- **FR-3.4** Plays `Sound.round()` or `Sound.match()` depending on which section
  triggered it.
- **FR-3.5** Transition must respect Reduced Motion (see FR-9.1).

### FR-4 Killfeed Toasts
- **FR-4.1** Fixed top-right stack of pixel-font toasts shaped like a game killfeed:
  `meet  [ELIM]  WEAPON  ▸  target`.
- **FR-4.2** Trigger condition: a `scroll` listener fires `maybeFire()` whenever the
  user has scrolled > 250 px since the last check.
- **FR-4.3** Rate limit: minimum 6s cooldown between fires; 25% probability per eligible
  event; pool entries are not reused until the entire pool has been consumed.
- **FR-4.4** Each toast auto-dismisses 3.5s after appearing.
- **FR-4.5** Content pool: at least 8 entries (currently 8), all dev-flavored
  (e.g. "meet [ELIM] TRY/CATCH ▸ unhandled exceptions").

### FR-5 Click-to-Ping
- **FR-5.1** Clicking on the background (any non-interactive element) places an
  expanding ring + center dot at the cursor that animates `scale 0.2 → 2.5, opacity
  1 → 0` over 1s.
- **FR-5.2** Ignored when the click target is inside `a / button / [role="button"] /
  input / textarea / select / [data-interactive]`.
- **FR-5.3** Plays `Sound.ping()` simultaneously.
- **FR-5.4** **Touch fallback:** ping still fires on `click` events (touch events
  synthesize clicks); cursor visual is irrelevant.

### FR-6 Scrolling System Log (Main Menu only)
- **FR-6.1** Fixed at bottom-left of the Main Menu, behind the menu content (`z-0`).
- **FR-6.2** Visible area shows up to 6 lines with a top + bottom mask-image gradient
  fade.
- **FR-6.3** New lines spawn on a 1.3s interval, fade in at the bottom, push the stack
  up, and the oldest line fades out at the top.
- **FR-6.4** Content pool: ≥ 20 entries (currently 22), mix of `[OK]` and `[..]`
  prefixes. `[..]` prefix is colored in accent, `[OK]` in foreground.
- **FR-6.5** Pool consumer must not pick the most recent line twice in a row.
- **FR-6.6** Component must `clearInterval` on unmount.

### FR-7 Background Birds Mini-Game (Main Menu only)
- **FR-7.1** Pixel-art birds (16×10 sprite, 2 flap frames + 1 dead frame) fly across the
  upper ~32 % of the viewport with randomized direction, y-position, and speed
  (60–140 px/sec).
- **FR-7.2** Spawn rate: first bird at 1.2s after mount, then a random delay of
  2.8–5.5s between spawns; max 3 birds alive simultaneously.
- **FR-7.3** Each bird owns its own `requestAnimationFrame` loop that handles both
  flying (`x += dir * speed * dt`, flap toggle every 220 ms) and death.
- **FR-7.4** **Click-to-kill:**
  - On click, the bird switches to a "dead" sprite (X-eyes) and the same RAF loop
    runs: `dy = ½ · 1600 · t²` (gravity), `rot = ±720° · t/0.8`, `opacity = 1 - t/0.8`.
  - After 0.85s the bird despawns.
  - Bird buttons carry `data-no-click-sound` so the global click sound is suppressed;
    `Sound.headshot()` plays instead.
- **FR-7.5** Kill feedback toast ("HEADSHOT" or "TARGET ELIMINATED") appears at the
  click point for 0.9s.
- **FR-7.6** Persistent `BIRDS NN` counter sits above the Main Menu bottom HUD bar.
- **FR-7.7** **Touch fallback:** birds still spawn and animate, but ergonomics on mobile
  are out of scope; clicking should still work via tap-to-click synthesis.

### FR-8 Sound System
- **FR-8.1** Implemented in pure Web Audio API (oscillators + envelope + sweep), no
  external audio files.
- **FR-8.2** Single AudioContext, single master gain at 0.12 — initialized lazily on
  first user gesture.
- **FR-8.3** Available sounds: `hover`, `click`, `ping`, `match`, `headshot`, `defeat`,
  `round`. (Defeat is currently unused but defined.)
- **FR-8.4** A global `SoundProvider` mounts capture-phase listeners that fire `hover`
  on first entry of any interactive element and `click` on any click — excluding
  elements with `[data-no-click-sound]`.
- **FR-8.5** Per-feature explicit calls:
  - PLAY button → `Sound.match()`.
  - Bird click → `Sound.headshot()`.
  - Ping → `Sound.ping()`.
  - Round transition → `Sound.round()` or `Sound.match()` depending on section.
- **FR-8.6** Mute state persisted to `localStorage` under key `meet:sound` and survives
  reload. Default ON for first-time visitors.

### FR-9 Settings
- **FR-9.1** **REDUCED MOTION toggle** must, when ON:
  - Disable Framer Motion section reveals, round-transition wipe, and ambient animations
    (glitch flicker, scanlines pulse).
  - Lenis must fall back to native scroll.
  - Killfeed toasts continue but with no entry/exit animation.
  - **Currently visual-only — see §8 issue I-1.**
- **FR-9.2** **SOUND FX toggle** must enable/disable all `Sound.*` calls and persist via
  `localStorage:meet:sound`. (Implemented.)
- **FR-9.3** Settings panel must be reachable from both the Main Menu and from the
  site shell (currently only from the Main Menu).

### FR-10 Project Detail Pages (`/agents/[slug]`)
Each project entry must contain:
- `slug` (URL segment) — kebab-case.
- `id` — display ID like `AGT-01`.
- `name` — short, all-caps, single word ideally.
- `role` — Valorant agent role (CONTROLLER / SENTINEL / DUELIST / INITIATOR).
- `hook` — one-line elevator pitch shown on the roster card.
- `desc` — 2–4 sentence narrative for the detail page.
- `palette` — accent hex color for that project's sprite.
- `year`, `status` (LIVE / BETA / INTERNAL / RETIRED).
- `repo` — GitHub URL or `null`.
- `live` — live URL or `null`.
- `stack` — array of `[name, proficiency 0–100]` tuples (3–4 items).
- `abilities` — exactly 4 entries of `[key, name, desc]` (Q / E / C / X).
- `highlights` — 3 short chips (e.g. `"12k DAU"`, `"99.95% uptime"`).

All project data lives in `lib/projects.js` and is the single source of truth.

### FR-11 Main Menu PLAY Flow
- **FR-11.1** PLAY → stage transitions to `matching` → `Sound.match()` plays → after
  1600ms `onPlay()` is called → parent unmounts MainMenu → site shell mounts with
  `IntersectionObserver` and Lenis initialized.
- **FR-11.2** Returning to the menu: header's "◀ MENU" button re-shows the
  MainMenu (`showMenu = true`).
- **FR-11.3** EXIT currently navigates to `https://www.google.com` — placeholder
  behavior; see I-2.

### FR-12 Site Shell
- **FR-12.1** Fixed top header showing "◀ MENU", "MEET", "// AGENT FILE", and faux
  HUD stats (HP / ARMOR / CREDS / ● LIVE).
- **FR-12.2** Footer line with build/patch identifier.
- **FR-12.3** Section dividers ("// CHECKPOINT 02", etc.) between each round.

## 6. Non-Functional Requirements

### NFR-1 Performance
- **NFR-1.1** Lighthouse Performance: ≥ 85 desktop, ≥ 70 mobile.
- **NFR-1.2** All UI animations use only `transform` and `opacity` (no width/height
  animations except the stat bars and the Main Menu loading bar, which are explicit
  exceptions because they're tiny and brief).
- **NFR-1.3** First Contentful Paint ≤ 1.5s on broadband desktop.
- **NFR-1.4** Total initial JS payload ≤ 250 kB gzipped at /.
- **NFR-1.5** Project detail pages must be statically generated (no SSR per request).
- **NFR-1.6** Birds, ping rings, and killfeed toasts must despawn from the DOM after
  their animation completes (memory hygiene).

### NFR-2 Accessibility
- **NFR-2.1** Respect `prefers-reduced-motion: reduce` in addition to the Settings
  toggle.
- **NFR-2.2** All informational text must meet WCAG AA contrast against `--bg`.
  (Currently fg is `#ECE8E1` and muted is `#6b6b6b`; the muted color on `#121212`
  must be verified.)
- **NFR-2.3** All interactive elements must remain keyboard-focusable; focus styles
  must be visible (a 1-px accent ring is acceptable).
- **NFR-2.4** Custom cursor must not break any pointer-input behavior; native cursor
  is restored on touch/coarse-pointer devices.
- **NFR-2.5** Birds + ping + killfeed are decorative — they MUST NOT be required for
  navigation or contact and MUST NOT be announced by screen readers
  (`aria-hidden="true"` on all decorative containers).
- **NFR-2.6** Semantic HTML for crawlability: each section uses `<section id="...">`,
  one `<h1>` per page (the agent detail uses the project name; the home page uses
  the MEET title), and `<nav>` / `<header>` / `<main>` / `<footer>` landmarks where
  appropriate.

### NFR-3 Browser / Device Support
- **NFR-3.1** Desktop first. Targets: latest Chrome, Firefox, Safari, Edge.
- **NFR-3.2** Mobile Safari + Chrome Android must render and be navigable. Cursor,
  birds, and ping have explicit fallbacks. Lenis smooth scroll must not break iOS
  rubber-banding.
- **NFR-3.3** No requirement to support IE or pre-2022 browsers.

### NFR-4 SEO
- **NFR-4.1** Per-page `<title>` and `<meta name="description">`. Home title is
  `MEET // AGENT PORTFOLIO`. Agent detail title is `${NAME} // AGENT FILE — MEET`.
- **NFR-4.2** Open Graph image must exist (currently missing — see I-15) at 1200×630,
  themed to the tactical HUD.
- **NFR-4.3** robots.txt and sitemap.xml must be generated (currently missing).
- **NFR-4.4** Canonical URLs declared on every route.

### 7.2 Voice & tone guidelines

- **Tactical-HUD-flavored, dry humor allowed.** "Eliminating bottlenecks before they
  spawn" is on-brand; "I am passionate about delivering value to stakeholders" is not.
- **No generic corporate language.** Avoid: "passionate", "synergy", "results-driven",
  "leverages", "best-in-class".
- **Concrete > abstract.** Use real numbers, real deploy counts, real outcomes wherever
  possible. The killfeed only works because it names a specific thing being killed.
- **Short sentences.** Pixel font + small caps make long sentences painful to read.
- **All caps for HUD labels (font-pixel); sentence case for prose (font-mono).**
- **Middle dot (·) is fine in mono, broken in pixel.** Press Start 2P doesn't ship
  many unicode glyphs — keep `.font-pixel` text to ASCII only.
